你现在是一名资深全栈架构师、产品经理、企业级 SaaS 工程负责人和 AI Agent 系统工程师。请根据以下产品定位，直接设计并实现一个可运行的企业级前后端应用系统。

项目名称暂定：ContentOS 中文创作者工作流系统

一、项目目标

我要做的不是一个单点 AI 工具，而是一套面向中文内容创业者、职业 IP、小品牌内容团队、小型 MCN 的“创作决策与执行工作台”。

系统需要形成两个核心闭环：

1. 洞察闭环：
   热点发现 → 热点聚类 → 趋势评分 → 选题生成 → 选题池管理

2. 执行闭环：
   人设记忆 → 内容生成 → 多平台适配 → 内容日历 → 发布计划 → 数据复盘 → 反哺人设和选题

请不要把它做成五个分散的聊天框，而是做成一个完整的企业级工作流系统。

核心 Agent 包括：

1. 热点 Agent
   - 输入：关键词、行业、竞品列表、目标平台、时间窗口
   - 输出：热点簇、早期信号、热度曲线、风险提示、建议切入角度
   - 工作流：数据采集 → 去重 → 聚类 → 热度/增速评分 → 语义摘要 → 告警
   - MVP 阶段可使用模拟数据、RSS、公开新闻源、手动导入数据，不要依赖非法爬虫
   - 未来要预留抖音、小红书、B 站、公众号、微博、快手等平台的官方授权数据接口

2. 选题 Agent
   - 输入：热点卡片、人设画像、历史爆文、商业目标、禁用话题
   - 输出：选题池、标题角度、内容提纲、发布节奏建议
   - 工作流：调取热点与历史素材 → 匹配人设 → 生成候选题 → 相关性与竞争度重排 → 输出内容日历

3. 人设 Agent
   - 不要做成一个孤立聊天框
   - 要沉入系统底层，成为“品牌记忆层”
   - 输入：历史稿件、创始人采访、品牌文档、评论区反馈、禁词、禁承诺列表
   - 输出：语气说明书、口头禅、高频词、禁忌表达、风格样例、目标受众版本
   - 必须支持可审阅、可编辑、可版本化
   - 要支持多品牌、多账号、多成员协作

4. 内容 Agent
   - 输入：选题 brief、人设档案、平台、体裁、素材包、目标 CTA
   - 输出：口播稿、图文稿、短标题、封面文案、分发版本、多平台改写稿
   - 工作流：结构化 brief → 初稿生成 → 风险审查 → 平台适配 → 人工审核 → 导出/发布计划
   - 必须支持不同平台的内容模板，例如：小红书图文、抖音口播、B 站视频脚本、公众号长文、视频号短内容

5. 数据分析 Agent
   - 输入：授权账号数据、作品表现、评论、转化目标、投放时间、发布时间、平台等上下文
   - 输出：周报、异常告警、涨跌解释、下轮选题建议、A/B 假设
   - MVP 可先做“轻量数据分析”，支持手动导入 CSV / Excel / JSON 数据
   - 未来预留官方 API 授权数据接入
   - 分析结果必须能回流到选题、人设和内容生成流程中

二、重要产品原则

1. 产品定位不是“全自动代运营机器人”
   正确定位是：
   基于授权数据、品牌记忆和多平台工作流的创作决策与执行系统。

2. 必须重视合规
   - 不做未经授权的大规模平台数据搬运
   - 不做绕过平台限制的爬虫
   - 所有平台数据来源必须抽象为合法数据源：官方 API、用户授权、用户上传、公开合规数据、第三方合法数据服务
   - 必须支持用户删除授权数据
   - 必须有审计日志
   - 必须有数据来源标记
   - 必须有内容风险提示
   - 避免默认面向医疗、金融、法律、时政等高风险内容场景

3. 首批目标用户
   - 成长型职业 IP：1–20 万粉，1–3 人团队
   - 小品牌内容团队：3–10 人
   - 小型 MCN / 代运营团队：多账号、多客户、多平台内容管理

4. MVP 重点能力
   第一阶段必须优先实现：
   - 热点看板
   - 选题池
   - 人设记忆层
   - 内容生成工作台
   - 内容日历
   - 轻量数据复盘
   - 团队与权限
   - 数据源管理
   - Agent 运行记录
   - 审计日志

九、AI Prompt 模板要求

   - 输出对应平台内容
   - 必须保持人设一致
   - 必须避免夸大承诺、侵权、违规表达

5. 内容风险检查 Prompt
   - 检查事实风险、版权风险、平台风险、敏感表达、过度承诺
   - 输出 riskLevel、riskItems、rewriteSuggestions

6. 数据分析 Prompt
   - 输入作品指标、账号基线、发布时间、内容主题、评论摘要
   - 输出涨跌解释、异常点、下轮建议、可实验假设

十五、验收标准

最终系统必须满足：

1. 可以本地启动
   - npm install
   - docker compose up -d
   - npx prisma migrate dev
   - npx prisma db seed
   - npm run dev

2. 访问首页后可以进入 Dashboard

3. 可以看到 mock 热点数据

4. 可以从热点生成选题

5. 可以把选题转成内容 brief

6. 可以基于人设生成内容草稿

7. 可以对内容进行风险检查

8. 可以把内容加入内容日历

9. 可以导入 mock 数据并生成数据分析报告

10. 可以查看 Agent 运行记录

11. 可以查看审计日志

12. 不同 workspace 的数据不能混用

13. TypeScript 不应有明显类型错误

14. 至少有基础测试通过

十六、执行方式

请不要只输出方案，不要只写伪代码，不要只写产品文档。

你要直接完成工程实现。

执行顺序建议：

第一步：检查当前仓库结构。如果为空，创建完整项目；如果已有项目，在不破坏现有结构的前提下集成。

第二步：建立数据库 schema、环境变量、Docker Compose、Prisma seed。

第三步：实现基础认证、workspace、RBAC、审计日志。

第四步：实现 Agent 框架和 Mock AI Provider。

第五步：实现热点、选题、人设、内容、数据分析核心服务。

第六步：实现前端页面和交互闭环。

第七步：补充测试、README、启动命令。

第八步：自检并修复类型错误、构建错误、明显运行错误。

十七、输出要求

完成后请输出：

1. 你创建或修改的核心文件列表
2. 本地启动命令
3. 已实现功能清单
4. 尚未接入但已预留的能力
5. 关键架构说明
6. 测试命令
7. 后续迭代建议

注意：
- 代码必须优先保证可运行
- 功能可以先 mock，但架构必须企业级可扩展
- 不要使用未经授权的平台爬虫
- 所有平台能力必须通过 adapter 抽象
- 所有 AI 能力必须通过 provider 抽象
- 所有重要业务操作必须写审计日志
- 所有 Agent 调用必须可追踪
- 所有生成内容必须经过风险检查或预留风险检查入口
- 用中文作为默认 UI 文案
